# 批量辅助抓取（书签 + 当前账号推文）

## What We're Building
为 twitter-web-exporter 增加一个“手动运行”的批量辅助抓取入口，放在设置面板中。用户可在一个入口里勾选需要执行的任务，然后由脚本按顺序自动完成抓取辅助流程。

第一阶段只覆盖两个任务：
- 抓取所有书签
- 抓取当前登录账号的个人推文历史（以 Web 端可加载范围为准）

这个能力的目标不是新增主动 API 拉取，而是在现有“拦截网页自身请求”的架构上，替代重复的人工操作。脚本负责自动进入对应页面、持续触发页面加载更多内容，并在过程中控制节奏，降低高频操作带来的风控风险。

## Why This Approach
我们在“自动辅助页面加载”和“主动模拟请求 GraphQL”之间，选择前者，原因是：
- 与当前仓库架构一致。现有实现通过拦截 XHR 响应采集数据，不直接请求 Twitter/X 接口。
- 风险更低。行为更接近人工浏览和滚动，而不是脚本批量直连接口。
- 更容易复用现有模块。`BookmarksModule` 和 `UserTweetsModule` 已经具备数据解析与入库能力，只缺少统一调度入口。

这意味着第一阶段解决的是“省去重复手动滚动和切页”，而不是突破平台可见范围限制。对于用户推文历史，抓取上限仍受 Twitter/X Web 端可加载深度约束，不能承诺获取账号全部历史推文。

## Key Decisions
- 触发入口：放在设置面板中，作为显式的手动工具入口
  - 理由：这是低频、长时运行操作，放在设置面板比放主面板更合适。

- 任务组织：单一入口，允许勾选要执行的任务
  - 理由：既支持“一次跑完”，也支持只运行书签或只运行我的推文。

- 抓取方式：自动辅助页面加载，不新增主动 API 请求
  - 理由：保持与现有拦截式架构一致，降低实现和风控复杂度。

- 执行顺序：串行执行多个任务，不并发
  - 理由：避免多个页面动作互相干扰，也降低单位时间内的请求密度。

- 节流策略：混合节流
  - 理由：短间隔持续滚动，累计一定步数后插入长暂停，更接近人工操作。

- 完成判定：以“多次无新增/无变化”作为主要停止条件，并设置硬上限兜底
  - 理由：既能在正常到底时自动停止，也能避免异常页面导致死循环。

- 推文范围：只抓当前登录账号的个人推文历史（按 Web 端可见范围）
  - 理由：需求聚焦于当前用户，且现有模式无法突破 Web 端天然可见限制。

- 风险提示：UI 需要明确说明频率控制与平台限制
  - 理由：防止用户误解为“完全无风险”或“能拿到全部历史数据”。

## Resolved Questions
- 入口位置：设置面板。
- 任务形态：一个入口，支持勾选多个任务。
- 节流模型：混合节流，不使用单一固定间隔。
- 停止条件：采用“多次无变化”加最大运行上限。
- 第一阶段范围：只做辅助抓取，不做主动接口拉取。
