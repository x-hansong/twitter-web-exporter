# Browser Migration Import Export

## What We're Building
为 `twitter-web-exporter` 增加“浏览器迁移”能力，让用户可以把当前浏览器中的本地数据库与应用配置打包导出，再在另一台浏览器中一次性导入恢复。

本次需求聚焦最小可用方案：以“单个迁移文件”作为核心交互，而不是分别处理数据库和配置。迁移包需要同时包含本地 IndexedDB 数据和 localStorage 中的完整配置快照，包括同步后端、访问密钥等敏感配置。导入时以“当前浏览器完整替换为迁移包内容”为准，不做合并。

## Why This Approach
我们比较了三种方向：

### Approach A: 单文件一键迁移包
把数据库和配置统一导出为一个迁移包，在新浏览器中选择文件后导入。

**Pros:**
- 最贴近“从浏览器 A 迁移到浏览器 B”的用户目标
- 流程最短，用户不需要理解数据库和配置的差异
- 更容易文档化和支持排障

**Cons:**
- 导入是高风险操作，需要明确提示覆盖语义
- 迁移包包含敏感配置，文件保管要求更高

**Best when:** 目标是完整迁移当前工作环境，而不是局部恢复

### Approach B: 数据库与配置分别导入导出
保留两个独立入口，让用户分别操作数据库和配置。

**Pros:**
- 灵活，适合局部恢复
- 对排障和高级用户更透明

**Cons:**
- 操作复杂，容易漏掉其中一部分
- 不符合“浏览器迁移”的主路径

**Best when:** 主要目标是高级维护，而不是普通迁移

### Approach C: 同时提供一键迁移与高级分离入口
默认提供单文件迁移，同时保留数据库和配置的独立入口。

**Pros:**
- 覆盖的使用场景最广
- 既照顾普通用户，也照顾高级用户

**Cons:**
- 首版产品面和文案复杂度都会增加
- 当前需求并不要求高级恢复能力

**Best when:** 已验证一键迁移稳定后，再扩展更多高级工具

最终选择 Approach A，因为它最符合当前动机，也最符合 YAGNI。我们先把“完整迁移”这条主路径做好，而不是一开始就扩展高级入口。

## Key Decisions
- 迁移对象：数据库与配置一起打包
  - 理由：用户目标是从一个浏览器迁移到另一个浏览器，而不是只迁移局部状态。

- 导出形态：单个迁移文件
  - 理由：降低操作成本，避免用户遗漏数据库或配置中的任意一部分。

- 配置范围：完整配置快照
  - 理由：需要包含主题、模块开关、同步配置、Supabase/MinIO 凭证、token 等全部配置，保证迁移后环境一致。

- 导入语义：覆盖当前浏览器中的数据库与配置
  - 理由：迁移场景下，“复制原环境”比“尝试合并”更清晰，也更不容易产生脏状态。

- 风险提示：导入前必须二次确认
  - 理由：导入会覆盖本地已有数据和配置，必须给用户明确的危险操作提醒。

- 首版范围：围绕浏览器迁移主流程，不优先做分离式数据库/配置导入导出
  - 理由：先完成核心价值，再根据反馈判断是否需要高级入口。

## Resolved Questions
- 主交互形态：单文件一键迁移包。
- 导入行为：覆盖当前浏览器中的数据库和配置。
- 配置是否包含敏感信息：包含。
- 风险交互：选择文件后先强提醒，再二次确认执行。
